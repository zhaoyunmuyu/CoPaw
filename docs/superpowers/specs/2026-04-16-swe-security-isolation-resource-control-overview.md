# Swe 安全、隔离与资源控制能力说明

## 1. 文档定位与适用范围

本文档面向研发管理与架构评审，说明 Swe 在安全治理、多租户隔离、资源控制以及运行审计方面的产品化能力。文档重点回答三个问题：

- Swe 当前已经具备哪些平台级控制能力
- 这些能力如何形成统一的治理闭环
- 这些能力适用于哪些场景，以及边界在哪里

本文档采用内部产品方案说明口径，不展开代码细节，不作为开发实现文档，也不作为主机安全或合规审计的替代材料。

## 2. 总体能力框架

Swe 不是将“安全”“隔离”“资源限制”分散挂在个别功能上，而是围绕多租户智能体平台构建了一套贯穿访问入口、租户上下文、工具执行、外部扩展、子进程运行、模型调用和运行留痕的控制体系。

从产品能力角度看，这套体系可以概括为三层：

### 2.1 访问与治理层

负责身份校验、敏感操作审批、工具调用治理，以及高风险动作的前置拦截。它解决的是“谁可以进来、谁可以做什么、哪些动作需要人工确认”的问题。

### 2.2 租户隔离层

负责将不同租户的工作目录、密钥、模型配置、会话数据、技能和记忆资产分离，避免跨租户数据污染、上下文串用和越权访问。它解决的是“不同租户之间如何稳定隔离”的问题。

### 2.3 资源控制层

负责限制平台执行面的资源消耗，包括模型调用并发与频率控制、子进程 CPU 与内存限制、命令执行超时和后台任务运行边界。它解决的是“系统如何在多租户和多任务场景下保持稳定可控”的问题。

三层能力并不是彼此独立的。一次请求从进入系统开始，会先绑定身份与租户，再进入对应租户上下文，在执行工具、调用模型或启动子进程时继续继承治理与资源策略，最终再回到运行记录与审计链路中。这意味着 Swe 的控制体系是贯通式的，而不是单点式的。

## 3. 安全治理能力

### 3.1 安全治理目标

Swe 的安全治理重点不是“禁止一切”，而是将高风险动作纳入可识别、可拦截、可审批、可追溯的治理流程。对智能体平台而言，真正的风险主要来自工具执行面和扩展能力面，因此平台将执行前控制放在核心位置，而不是仅依赖执行后的补救。

### 3.2 访问认证与入口保护

平台支持在服务入口侧启用身份认证能力，对控制台和 API 访问形成基础门槛。其目标不是构建复杂的 IAM 体系，而是确保平台在对外暴露时具备最基本的登录认证和会话校验能力，避免未授权访问直接进入智能体执行面。

对内产品方案层面，这项能力的价值在于：

- 为控制台和接口提供基础准入控制
- 让智能体执行环境与公开访问面之间形成第一道隔离
- 为后续审批、审计和租户识别提供身份基础

### 3.3 敏感工具前置治理

Swe 将高风险工具调用纳入统一治理范围，典型包括 Shell 执行、文件读写、浏览器访问等具有外部影响能力的操作。平台会在执行前对工具参数和命令意图进行检查，识别破坏性命令、权限提升、危险网络执行、系统篡改、敏感资产访问等高风险模式。

这项能力的产品价值在于：

- 将“Agent 可能产生危险动作”的风险前移到执行前治理
- 让高风险操作从不可控行为转化为可规则化处理的受控行为
- 为企业场景提供可配置的安全基线，而不是完全依赖模型自觉

### 3.4 审批驱动的风险释放机制

对于命中高风险规则但业务上可能仍需执行的动作，Swe 提供人工审批机制。也就是说，平台并不简单地在“全部放行”和“全部阻断”之间二选一，而是把高风险执行纳入人工确认链路。

这项机制的价值主要体现在三个方面：

- 在安全性和可用性之间取得平衡
- 适配需要人工兜底的企业流程
- 让高风险动作具备明确责任归属和处置记录

从产品能力角度看，审批机制使 Swe 更适合进入真实业务环境，而不仅是实验性智能体场景。

### 3.5 敏感资产访问保护

平台对密钥、认证数据、敏感目录以及高敏感配置资产保持严格约束，避免智能体借由文件工具或命令执行直接访问系统敏感面。其核心思路是，即使 Agent 具备文件与命令能力，其可达范围也必须被平台控制，而不是自然扩展到宿主机敏感资产。

这项能力带来的平台价值包括：

- 限制智能体对平台关键密钥和配置的暴露面
- 降低误读、误改、误删除敏感资产的风险
- 将“执行能力”与“系统级权限”明确分离

### 3.6 扩展内容风险控制

Swe 不只治理运行时工具调用，也治理进入系统的扩展内容。对于技能、脚本和附带资源，平台具备扫描与筛查能力，用于降低危险逻辑、异常文件结构或不受控扩展内容进入运行环境的概率。

这一点对平台长期运营尤为重要，因为扩展机制通常是能力增长最快的部分，也是最容易成为治理盲区的部分。将扩展内容纳入治理，意味着平台不仅关注“当前执行什么”，也关注“系统未来会获得什么能力”。

## 4. 多租户隔离能力

### 4.1 隔离目标

Swe 的隔离体系围绕“请求级租户绑定”和“租户级资产分层”展开，目标是确保不同租户在身份、目录、密钥、模型配置、会话状态和工作空间上的运行边界稳定可控。

这意味着 Swe 的隔离不是单一的目录隔离，而是从请求入口到运行上下文再到持久化资产的整链路隔离。

### 4.2 请求级租户身份绑定

平台在请求进入时会解析并校验租户身份，将租户与用户信息绑定到当前请求上下文中。只有经过校验的租户请求，才会进入后续工作区装配、工具执行和模型调用链路。

这一机制的产品意义在于：

- 将租户识别前移到入口层
- 避免执行过程中临时推断租户导致的上下文混乱
- 为后续所有隔离能力提供统一依据

在架构评审视角下，这代表平台采用的是“先识别、再执行”的隔离模型，而不是“执行中附带租户参数”的松散模型。

### 4.3 工作区与运行上下文隔离

每个租户拥有独立的工作目录和工作区上下文。平台在处理请求时，会为当前租户装配对应的工作空间，并将后续文件工具、会话管理、任务执行等行为约束在该租户范围内。

其核心价值体现在：

- 不同租户之间的工作文件和运行状态彼此独立
- 平台能够在租户维度进行懒加载、初始化和运行管理
- 后续工具执行天然继承租户边界，而不是依赖业务方额外传参

这使 Swe 更适合承载多用户或多团队并行使用的智能体场景。

### 4.4 数据与资产分层隔离

Swe 将租户数据按资产类型分层管理。配置、会话、记忆、模型、技能、媒体和工作文件等资产各自按租户维度存放，避免多租户共享同一份运行态数据或状态文件。

从平台能力角度看，这种分层隔离有两个重要价值：

- 防止不同类型资产在同一目录下混用，降低串扰风险
- 为后续迁移、备份、清理、审计和按租户运维提供更清晰的边界

这不是简单的“一个租户一个文件夹”，而是面向平台治理的资产分类隔离。

### 4.5 密钥与模型配置隔离

Swe 将租户的模型接入信息、密钥和激活模型配置按租户独立存放。不同租户可以拥有不同的 Provider 配置、模型槽位与默认模型，不共享运行密钥，也不共享激活状态。

这项能力对于企业级多租户平台尤其关键，其价值包括：

- 避免租户之间共用模型凭据
- 支持不同租户采用不同模型供应策略
- 让模型接入成为租户级能力，而不是平台全局单例能力

这也意味着平台在“模型调用能力”这一层面实现了真正的租户隔离，而不是只在会话层做表面隔离。

### 4.6 路径边界与越界防护

为了防止文件工具和命令执行借由相对路径、绝对路径或符号链接逃逸租户范围，Swe 在本地路径访问侧建立了明确的路径边界控制。平台会将工具可访问路径约束在当前租户工作空间内，防止智能体跨出本租户边界访问其他租户资产或宿主机不应开放的区域。

这项能力的产品价值非常直接：

- 降低路径穿越和误操作扩散风险
- 保证“租户隔离”不只体现在目录规划上，也体现在执行校验上
- 为文件工具、Shell 工具等高风险能力提供基础保护面

### 4.7 新租户初始化与继承机制

当新租户首次进入系统时，平台具备按租户初始化运行空间的能力，包括基础目录、默认工作区、基础配置和必要资产的准备。同时，平台可以继承默认租户的一部分基础配置，降低新租户接入成本。

从产品方案角度看，这意味着 Swe 的多租户能力不是“手工创建目录”的工程化补丁，而是具备可运营、可复制、可扩展的租户生命周期起点。

## 5. 资源控制能力

### 5.1 资源控制目标

Swe 的资源控制目标是避免单次请求、单个租户或单个执行动作对系统稳定性造成失控影响。控制重点覆盖三个方面：

- 模型调用资源
- 子进程执行资源
- 任务运行时边界

资源控制不是附属优化项，而是多租户智能体平台可长期稳定运行的基础能力。

### 5.2 模型调用节流与并发控制

平台对大模型调用具备统一的节流能力，通过并发控制、单位时间请求量控制和触发限流后的全局冷却机制，减少模型服务被突发流量打爆或反复触发上游限流的概率。

这项能力带来的价值包括：

- 平滑高并发请求带来的模型访问压力
- 减少上游模型接口频繁返回限流错误
- 在多租户共享模型资源的情况下提升平台整体稳定性

从产品角度看，这是一种“平台级流量缓冲器”能力，使模型调用不再完全暴露在瞬时业务波动之下。

### 5.3 子进程 CPU 与内存限制

针对 Shell 执行和部分外部标准输入输出型进程，Swe 可以在启动时施加 CPU 时间和内存上限，防止单个命令或外部进程持续吞噬宿主资源。

这一能力特别适合以下场景：

- 智能体需要调用本地命令进行文件处理或构建操作
- 平台需要接入外部进程型扩展能力
- 希望将外部执行带来的资源风险约束在可接受范围内

从平台稳定性角度看，这项能力把“命令执行成功与否”之外的另一个核心问题纳入治理，即“即便命令异常，也不能无边界消耗资源”。

### 5.4 执行超时控制

除了 CPU 和内存限制外，Swe 还在命令执行层提供了运行时超时约束，用于限制单个命令的 wall-clock 生命周期。这样可以覆盖那些 CPU 消耗不高但长时间阻塞、等待 I/O 或挂起不退出的执行场景。

这项能力的价值在于：

- 对慢阻塞命令形成兜底控制
- 避免任务被单个低活跃度命令长期占用
- 与 CPU 时间限制形成互补，覆盖更完整的资源风险面

### 5.5 多任务运行中的资源平滑

Swe 在后台任务和定时任务链路中，同样沿用租户上下文和执行限制逻辑。这意味着资源控制不是只对交互式会话生效，而是会随着任务执行链路继续传递，减少离线任务成为控制盲区的风险。

对于研发管理和架构评审而言，这一点很重要，因为很多平台问题并不是发生在即时对话，而是发生在批处理、定时任务或自动化后台流程中。

### 5.6 使用量记录与成本感知

平台能够对模型 Token 使用情况进行记录与聚合，用于支撑成本分析、用量观察和后续配额设计。虽然这类能力不直接阻断资源消耗，但它为资源治理提供了数据基础，是从“可控执行”走向“可控成本”的前提。

这项能力的产品价值主要在于：

- 为模型成本管理提供事实基础
- 支持按模型、按时间、按租户场景做使用分析
- 为后续预算、额度和精细化运营提供可扩展支点

## 6. 可观测与审计支撑

### 6.1 运行追踪能力

Swe 具备对模型调用、工具使用和会话过程进行运行追踪的能力，用于还原一次请求在平台中的执行路径。对于智能体平台而言，这类追踪能力不仅是性能诊断工具，也是一种治理支撑能力，因为它能帮助平台回答“这次动作是谁触发的、调用了什么、经历了哪些环节”。

### 6.2 敏感信息脱敏

平台在运行记录侧具备基础脱敏能力，会对常见密钥类字段和敏感内容进行隐藏或裁剪，避免追踪与日志系统自身成为新的泄露面。

其价值在于：

- 平衡可观测性与数据暴露风险
- 让运行记录更适合进入内部排障与分析流程
- 降低审计链路对敏感字段的二次传播

### 6.3 审批与执行记录

对于进入人工审批流程的高风险动作，平台会保留审批状态和处理结果，形成从风险发现到人工放行或拒绝的治理闭环。这使审批机制不仅是一个交互动作，也成为可回溯的运行记录。

### 6.4 服务状态与运行支撑

平台同时具备服务心跳、实例状态和部分后台作业运行信息的管理能力，可用于辅助判断平台是否稳定运行、任务是否正常执行、实例是否处于健康状态。

从架构评审视角看，这些能力共同构成了安全和资源控制体系的“观测面”。没有观测，控制就难以验证；没有记录，治理就难以沉淀。

## 7. 能力边界说明

为了避免对平台能力形成过度解读，以下边界需要在内部方案说明中明确：

### 7.1 平台治理不等于主机级全面防护

Swe 当前提供的是智能体平台执行治理能力，重点在访问控制、执行前检查、租户隔离、审批放行和资源约束。它并不替代主机侧安全加固、容器隔离、EDR、防火墙、主机审计或零信任接入体系。

### 7.2 资源限制覆盖重点是平台外部执行面

当前资源控制能力主要覆盖模型调用节流、命令执行超时，以及特定类型子进程的 CPU 和内存限制。它不是对平台内所有进程和所有组件的一刀切统一资源沙箱。

### 7.3 部分底层资源限制依赖运行平台能力

进程级资源限制的有效性与底层运行环境相关，在更适合相关能力的操作系统和部署环境中效果更稳定。因此，资源治理需要与实际部署环境联合评估，而不能只看平台开关本身。

### 7.4 审批机制是高风险动作的治理补强，不是业务流程系统替代

Swe 的审批机制用于高风险执行放行，不等同于企业完整流程审批系统。它适合作为智能体执行面的安全闸门，而不是取代更广义的组织审批平台。

### 7.5 当前能力强调“受控执行”，不承诺“绝对隔离”

平台已经在请求、目录、配置、工具访问和部分执行资源上形成较完整的隔离与控制体系，但对于真正高对抗场景，仍需结合容器、虚拟化、网络隔离和外部安全基础设施形成更高等级方案。

## 8. 总结

从内部产品方案角度看，Swe 已经形成了一套相对完整的智能体平台控制基座，其核心特点不是单点功能多，而是控制链条比较完整：

- 在访问入口侧，具备基础认证与租户识别能力
- 在执行治理侧，具备高风险工具前置检查、审批放行和敏感资产保护能力
- 在租户隔离侧，具备工作区、配置、密钥、模型和运行上下文的分层隔离能力
- 在资源控制侧，具备模型调用节流、并发控制、命令超时和部分子进程资源限制能力
- 在运行支撑侧，具备追踪、脱敏、审批记录和用量观察能力

这使 Swe 不只是“能运行智能体”的执行平台，而是已经具备一定企业化治理特征的多租户智能体平台。对于研发管理和架构评审而言，可以将其理解为一套以执行治理为中心、以租户隔离为底座、以资源控制和观测支撑为保障的产品能力体系。
