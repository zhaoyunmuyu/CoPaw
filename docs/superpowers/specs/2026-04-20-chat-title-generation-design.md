# 聊天记录标题异步摘要生成设计

> 设计日期: 2026-04-20
> 状态: 待评审

## 1. 背景与目标

当前聊天记录标题在会话首次创建时，直接取首条用户消息文本的前几位作为 `ChatSpec.name`。这有两个明显问题：

1. 标题可读性差，往往只是截断后的提问片段。
2. 标题生成逻辑分散在不同入口，console 与通用 channel 的截断长度还不一致。

本次目标是将聊天标题改为：

- 基于首条用户消息内容，由模型生成一个简短、可读的标题
- 采用异步方式，不增加首轮回复的主路径延迟
- 由后端统一生成，所有会话入口共享同一套规则
- 不覆盖用户手动重命名的标题

### 1.1 本期范围

**本期实现：**

- 后端统一异步生成聊天标题
- 首条用户消息作为标题摘要输入
- 仅在新建会话且仍为默认标题时触发生成
- 标题生成结果回写 `ChatSpec.name`
- 前端通过现有会话刷新链路显示最新标题
- 为标题生成、并发保护和回退行为补充测试

**本期不实现：**

- 标题失败后的自动重试队列
- 多轮消息后二次重写标题
- 前端本地标题生成或前端直接调模型
- 专门的 SSE/WebSocket 标题更新事件

---

## 2. 当前问题与约束

### 2.1 当前链路

当前标题主要在首次建会话时生成：

- `src/swe/app/routers/console.py`
- `src/swe/app/channels/base.py`
- `src/swe/app/runner/runner.py`

这些入口会在调用 `chat_manager.get_or_create_chat(...)` 前先构造一个 `name`，通常是首条消息文本截断后的结果。`ChatManager` 仅在首次创建会话时写入该名称，后续相同 `session_id` 的消息不会更新标题。

### 2.2 必须满足的约束

- 标题生成不能阻塞主对话回复
- 标题生成失败不能影响主流程
- 不得覆盖用户通过前端或 CLI 主动修改的标题
- 不要求前端理解标题生成规则，前端只负责显示
- 多入口必须行为一致，避免 console / channel / runner 再各自维护一套截断逻辑

---

## 3. 方案选择

### 3.1 候选方案

#### 方案 A：后端创建会话后异步生成标题

做法：

- 新会话先以默认标题创建
- 创建完成后在后端启动异步标题生成任务
- 任务完成后回写 `ChatSpec.name`

优点：

- 不阻塞首轮回复
- 逻辑集中在后端，适用于 console、runner、channel
- 前端只需消费 `chat.name`

缺点：

- 需要处理并发覆盖和刷新时机

#### 方案 B：前端在首轮消息后发起标题生成

优点：

- 交互链路直观

缺点：

- 标题逻辑散落到前端
- 其他入口无法复用
- 前后端容易出现标题规则不一致

#### 方案 C：把标题生成塞进主 runner 流程尾部

优点：

- 看似改动集中

缺点：

- 标题职责和主对话执行耦合过深
- 不利于覆盖所有入口

### 3.2 结论

采用 **方案 A：后端统一异步生成标题**。

该方案最符合本次目标：既保证用户首轮响应体验，又将标题生成规则收敛为一处后端能力。

---

## 4. 目标架构

### 4.1 模块划分

新增一个标题生成服务，职责单一：

```text
首条用户消息
  ↓
ChatManager 创建新 ChatSpec（默认标题）
  ↓
异步调度标题生成服务
  ↓
模型生成简短标题
  ↓
服务清洗/截断结果
  ↓
二次检查标题仍为默认名
  ↓
update_chat(...) 回写 ChatSpec.name
```

建议新增模块：

```text
src/swe/app/runner/title_generator.py
```

职责：

- 构造标题摘要 prompt
- 调用模型生成标题
- 清洗模型返回结果
- 执行“默认标题保护”与“手动重命名保护”

### 4.2 现有模块职责调整

#### `ChatManager`

- `get_or_create_chat(...)` 保留“首次创建才写入 name”的语义
- 入口侧不再传入截断后的首条消息作为最终标题
- 新建会话时统一使用默认标题

#### 会话入口

- `console.py`
- `channels/base.py`
- `runner.py`

这些入口仍然负责拿到首条用户文本，但不再把它当作最终标题，而是作为“候选摘要输入”传递给标题生成调度逻辑。

#### 前端 Chat 页面

- 继续只使用 `chat.name`
- 在会话仍为默认标题时，允许通过现有刷新链路更快拿到异步更新后的标题
- 不引入本地摘要逻辑

## 5. 后端详细设计

### 5.1 默认标题策略

新增常量，例如：

```python
DEFAULT_CHAT_NAME = "New Chat"
```

规则：

- 所有新会话初始标题统一为默认值
- 异步标题生成只对当前标题仍等于默认值的会话生效

这样可以把“是否允许回写标题”的判断标准收敛成一个简单、稳定的条件。

### 5.2 标题生成触发时机

触发条件：

1. 当前请求导致创建了一个新会话
2. 存在可提取的首条用户文本
3. 标题当前仍为默认值

非触发场景：

- 已存在的会话继续聊天
- 纯媒体内容且没有可用于摘要的文本
- 用户已手动重命名

### 5.3 异步执行方式

建议使用应用内异步任务调度，避免引入额外队列设施。

做法：

- 在首次创建会话成功后，用 `asyncio.create_task(...)` 或等价方式调度后台任务
- 任务内部自行捕获异常并记录日志
- 不向主对话链路回传标题生成结果

理由：

- 本期不需要复杂重试和持久化队列
- 目标只是“轻量增强”，失败直接回退默认标题即可

### 5.4 标题生成 prompt 约束

标题生成 prompt 必须强约束，避免模型输出解释性文字。

要求：

- 输出单行短标题
- 不加引号
- 不加句号
- 不使用“帮你”“关于”“请问”等低信息量词
- 不重复用户原句中的冗余铺垫
- 聚焦任务或主题本身

示例目标：

- 用户消息：`帮我分析一下 2024 年英伟达财报和市场预期差异`
- 目标标题：`英伟达财报预期差异分析`

### 5.5 结果清洗规则

模型返回后，服务端必须做二次清洗：

- 去首尾空白
- 只取第一行
- 去掉首尾引号、句号等包裹字符
- 空结果回退默认标题
- 超长时按统一上限截断，例如 24 到 32 个字符

服务端清洗是必要的，因为即使 prompt 严格，模型也不能保证百分之百遵守。

### 5.6 手动重命名保护

这是本次设计的核心保护逻辑。

异步任务在回写前必须再次读取最新 `ChatSpec`，并检查：

1. chat 仍然存在
2. `chat.name` 仍为默认标题

只有同时满足时才允许执行 `update_chat(...)`。

这样可以避免以下竞争：

- 用户在后台任务完成前手动改名
- 其他逻辑已提前写入更合理的标题

### 5.7 失败与降级

失败场景包括：

- 模型调用失败
- 超时
- 返回空文本
- 标题清洗后无有效内容

降级策略：

- 保留默认标题
- 记录 debug / warning 日志
- 不抛出到主链路
- 不影响聊天响应

## 6. 前端设计

### 6.1 展示原则

前端不参与标题生成，只展示后端返回的 `chat.name`。

这意味着：

- 默认展示仍可能短暂出现“新会话”
- 后台标题生成完成后，通过会话列表刷新拿到新标题

### 6.2 刷新策略

本期不新增专门推送事件，沿用现有会话拉取链路。

建议在 Chat 页面增加轻量优化：

- 当当前会话标题仍为默认值时
- 在首轮回复完成后的短时间窗口内
- 再主动刷新一次 `getSessionList()` 或当前 session

目标是缩短用户看到最终标题的时间，但不要求实时推送。

### 6.3 不采用的前端方案

不在前端做以下事情：

- 本地根据用户消息拼标题
- 前端单独调模型生成标题
- 为标题状态新增复杂 UI

因为这些都会让标题来源不再单一，增加一致性风险。

## 7. 测试设计

### 7.1 单元测试

需要覆盖以下最小行为：

1. 标题生成服务收到正常文本时，能输出清洗后的标题
2. 模型返回多行、带引号或带句号时，结果会被清洗
3. 模型返回空内容时，回退默认标题
4. 新建会话时会触发异步标题任务
5. 已存在会话不会重复触发标题任务
6. 用户已手动重命名时，异步任务不会覆盖名称

### 7.2 API / 集成测试

建议至少覆盖一条主链路：

- 发送首轮消息
- 会话先以默认标题创建
- 异步标题任务完成
- 再次读取 chat 列表或 chat 详情能看到新标题

### 7.3 非目标测试

本期不要求验证复杂重试或分布式队列行为，因为设计中没有引入这些机制。

## 8. 实施顺序建议

2. 为标题清洗规则编写测试
3. 实现标题生成服务与模型调用封装
4. 在会话首次创建链路中接入异步调度
5. 增加“默认标题保护”和“手动重命名保护”测试
6. 补前端轻量刷新优化
7. 运行相关测试并验证实际交互

## 9. 风险与后续演进

### 9.1 已知风险

- 异步任务如果依赖当前请求上下文中的模型配置，需要确认脱离主请求后仍可正常调用
- 如果会话列表刷新时机太弱，用户可能会较晚看到最终标题
- 默认标题与已生成标题只用字符串比较，需要确保默认值常量在各处统一

### 9.2 后续可扩展方向

- 为标题生成失败增加有限重试
- 标题生成完成后推送前端更新事件
- 对无文本消息设计更合适的兜底标题
- 对不同语言输入采用不同标题长度策略

---

## 10. 结论

本方案通过“后端统一异步生成 + 默认标题保护 + 手动重命名保护”的组合，替代现有首条消息截断标题逻辑。

该设计的核心价值是：

- 标题质量显著提升
- 首轮对话体验不受阻塞
- 各入口行为统一
- 不引入前端和多入口的重复实现

在不增加系统复杂度太多的前提下，这是一条风险可控、收益明确的演进路径。
